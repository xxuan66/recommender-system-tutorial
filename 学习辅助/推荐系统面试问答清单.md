# 推荐系统面试问答清单

> 这份文档不追求覆盖所有题目，而是围绕当前仓库的学习主线，整理一份更适合入门到初级进阶阶段使用的问答清单。

---

## 1. 怎么使用这份问答

更推荐把这份文档当成 3 种用途：

1. 学完一个章节后，检查自己能不能用自己的话回答。
2. 做完 MovieLens 实验后，检查自己能不能把实验讲清楚。
3. 做完 `16-电影推荐系统` 后，检查自己能不能把项目讲成一个完整故事。

如果一个问题你只能背定义，不能结合实验或项目来回答，通常说明理解还不够稳。

## 2. 基础认知问题

### Q1：推荐系统和搜索有什么区别？

推荐系统更强调“主动分发”，搜索更强调“响应明确需求”。

推荐系统通常要在用户没有明确输入时，预测他可能感兴趣的内容；搜索则更多是在用户给出 query 后返回相关结果。

### Q2：推荐系统的核心链路通常有哪些环节？

一个常见答案是：

`召回 -> 排序 -> 重排 -> 展示 -> 反馈回流`

如果是入门项目，也可以先从“数据 -> 模型 -> 评估 -> 推荐结果”这条更简单的主线讲起。

### Q3：为什么推荐系统不能只看准确率？

因为推荐系统不只是分类问题。很多真实任务更关心排序质量、覆盖率、多样性、长期留存，而不只是单点预测是否准确。

## 3. 经典算法问题

### Q4：UserCF 和 ItemCF 有什么区别？

UserCF 是找相似用户，再看相似用户喜欢什么；ItemCF 是找相似物品，再根据用户历史喜欢的物品做扩展推荐。

更进一步的回答可以补一句：

- UserCF 更像“和你相似的人也喜欢”。
- ItemCF 更像“喜欢这个的人也喜欢那个”。

### Q5：什么时候更适合用 ItemCF？

当物品关系相对稳定、用户规模较大、同时需要更强可解释性时，ItemCF 往往更合适。

### Q6：协同过滤最大的局限是什么？

它依赖历史交互数据，容易受数据稀疏、冷启动、热门偏置影响。当用户或物品交互太少时，相似度很难稳定。

### Q7：矩阵分解比协同过滤多解决了什么？

矩阵分解不只依赖显式相似度，而是学习用户和物品的隐向量表示，因此通常更擅长从整体结构中提取模式，也更容易衔接后续更复杂的推荐模型。

### Q8：为什么 FunkSVD 比传统 SVD 更适合推荐系统？

因为推荐系统数据天然稀疏，FunkSVD 直接优化已知评分，而不是先把整个矩阵强行填满再分解，所以更贴近推荐任务本身。

## 4. 评估问题

### Q9：RMSE 和 Precision@K 分别反映什么？

- `RMSE` 更偏评分预测误差。
- `Precision@K` 更偏前 K 个推荐结果里有多少是命中的。

推荐系统常常不能只看 RMSE，因为用户最终看到的是 Top-N 列表，而不是模型内部的回归误差。

### Q10：为什么离线指标好，不代表线上效果一定好？

因为离线数据分布、曝光机制、用户反馈方式，都可能和真实线上环境不同。离线实验更像筛选方案，不能完全替代线上验证。

### Q11：做最小实验时，至少该记录哪些东西？

至少记录：

- 数据集版本和划分方式
- 模型和关键参数
- 指标结果
- 1-2 组推荐样例
- 对结果的简单解释

## 5. 项目问题

### Q12：如果让你介绍一个电影推荐系统项目，你会怎么讲？

一个比较稳的结构是：

1. 先讲项目目标。
2. 再讲数据来源和处理方式。
3. 再讲 baseline 和模型选择。
4. 再讲离线评估结果。
5. 最后讲 Web 展示、推荐解释和下一步优化。

### Q13：一个推荐项目为什么不能只放代码？

因为项目能力不只是“能运行”。别人还需要看出你是否知道：

- 问题是什么
- 为什么这样设计
- 效果怎么评估
- 下一步怎么优化

### Q14：做作品集项目时，最少该具备什么？

至少要有：

- README
- baseline 对比
- 指标结果
- 推荐样例
- 下一步 roadmap

### Q15：如果用户历史行为很少，你怎么处理？

可以从几个方向回答：

- 热门推荐作为保底策略
- 内容特征辅助推荐
- 新用户引导收集偏好
- 混合推荐或冷启动策略

## 6. 追问型问题

### Q16：如果你现在的 baseline 效果一般，先改什么？

一个稳妥回答通常不是“立刻换更复杂模型”，而是先看：

1. 数据划分和评估方式是否合理。
2. 当前相似度、近邻数或矩阵分解参数是否需要调整。
3. 结果是否受热门偏置和数据稀疏影响。

### Q17：如果资源有限，你优先补模型、工程还是策略？

一个较成熟的回答通常会说：先看当前瓶颈在哪。

- 如果 baseline 都没立住，先补模型和实验。
- 如果实验结果不错但项目很难落地，先补工程。
- 如果模型稳定但业务指标不理想，先补策略。

### Q18：学完这套教程后，下一步怎么继续进阶？

可以按 3 条路线回答：

- 算法路线：继续补模型、论文和实验分析。
- 工程路线：继续补服务化、监控、日志和系统架构。
- 业务路线：继续补目标设计、A/B 测试和推荐策略。

---

## 7. 自查标准

如果你准备面试或汇报，更推荐最后用这 4 条做自查：

1. 我能不能不用照着文档，把问题讲清楚。
2. 我能不能把回答和自己的实验或项目连起来。
3. 我能不能说明当前方案的优点和局限。
4. 我能不能说出下一步优化方向，而不是只说“换更复杂模型”。

只要这 4 条能做到，大多数基础到初级进阶阶段的推荐系统交流就会顺很多。
