# MovieLens 实验说明与常见坑

> 这份文档不是单独讲新算法，而是给当前仓库里的经典推荐实验和项目实战提供一个统一的最小实验说明。

---

## 1. 这份文档解决什么问题

很多读者在看完 `05-协同过滤算法`、`07-矩阵分解` 和 `16-电影推荐系统` 后，会遇到同一个问题：

> 我知道每一章在讲什么，但真正开始做 MovieLens 实验时，不知道该先做什么、该记录什么、怎么判断结果是否靠谱。

这份文档就是为了解决这个问题。

它的目标不是把实验做得很复杂，而是先帮你建立一个可复现、可比较、可复盘的最小实验闭环。

---

## 2. 最小实验闭环

更推荐你按下面顺序做，而不是一开始就同时尝试很多模型：

1. 先固定数据集版本和划分方式。
2. 先跑一个协同过滤 baseline。
3. 再跑一个矩阵分解 baseline。
4. 记录指标、参数和样例推荐结果。
5. 最后再决定要不要接 Web 项目或更复杂模型。

一个最小闭环至少要留下这 4 类结果：

| 结果 | 最低要求 |
|------|----------|
| 数据说明 | 用的是哪个 MovieLens 版本，怎么划分训练集和测试集 |
| 指标结果 | 至少保留 RMSE 和一个 Top-N 指标 |
| 参数记录 | 至少记录 `k` 或 `n_factors`、`reg`、`epochs` |
| 推荐样例 | 至少展示 1-2 个用户的 Top-N 结果 |

---

## 3. 推荐的实验顺序

### 第一步：先跑协同过滤 baseline

建议先从下面二选一开始：

- `UserCF`
- `ItemCF`

这一步的重点不是追求最高分，而是先确认：

- 数据能顺利读进来。
- 训练和测试划分没有问题。
- 推荐结果能正常输出。
- 你知道当前 baseline 的局限在哪里。

### 第二步：再跑矩阵分解 baseline

建议优先从 `FunkSVD` 或 Surprise 中的 `SVD` 开始。

这一轮最重要的是回答两个问题：

1. 矩阵分解相对协同过滤有没有稳定提升。
2. 这种提升主要体现在哪些指标上。

### 第三步：保留推荐样例

不要只保留一张分数表。

更推荐额外保留这种信息：

```text
user_id: 42
history: Toy Story, Jumanji, Jurassic Park
top_recommendations:
- Star Wars
- Men in Black
- Back to the Future
```

它的价值是帮助你检查：

- 推荐是否明显跑偏。
- 模型是否只是偏向热门内容。
- 结果是否已经有基本可解释性。

---

## 4. 推荐记录模板

建议每次实验至少记下面这张表：

| 模型 | 核心参数 | RMSE | Precision@10 | Recall@10 | 结论 |
|------|----------|------|--------------|-----------|------|
| UserCF | `k=40` | - | - | - | baseline，可解释但对稀疏敏感 |
| ItemCF | `k=40` | - | - | - | 物品关系更稳定 |
| SVD / FunkSVD | `n_factors=50, reg=0.02` | - | - | - | 表达能力更强，需调参 |

如果你之后要写博客、README 或项目复盘，这张表会非常有用。

---

## 5. 最容易踩的坑

### 5.1 把未评分当成负反馈

在 MovieLens 里，很多 `0` 或空白更适合被理解为“未知”，而不是“明确不喜欢”。

### 5.2 只看 RMSE

RMSE 低，不代表 Top-N 推荐一定更好。推荐系统更接近排序问题，而不是单纯回归问题。

### 5.3 数据划分方式不固定

如果每次实验的数据划分都不一致，很多结果很难直接比较。

### 5.4 只保留最优结果，不记录过程

如果你只记住了“最好的一组参数”，以后很难复盘为什么这组参数更好。

### 5.5 没有留推荐样例

没有样例时，你只能看到数字，无法快速发现推荐方向是否明显异常。

---

## 6. 什么时候可以继续往项目走

如果你已经满足下面这些条件，就很适合继续进入 `16-电影推荐系统`：

1. 至少完成了一组 baseline 对比。
2. 至少记录了一组稳定的离线指标。
3. 至少保留了 1-2 个用户的推荐样例。
4. 能说清楚当前模型的优点、问题和下一步改进方向。

到这一步，你再做 Web 展示、推荐解释或作品集整理，会更扎实。
